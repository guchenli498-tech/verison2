## 2. 项目背景与建设目标

### 2.1 背景说明

本项目面向“徽派建筑文化传播与数字展示”场景，通过可视化手段构建沉浸式、可交互、可演示的网页系统。项目围绕“聚落格局 -> 构件体系 -> 三雕文化”形成渐进叙事链路，强调知识表达与展示体验统一。

### 2.2 建设目标

1. **文化传播目标**：将传统建筑知识转化为可浏览、可理解、可比较的可视化内容。  
2. **展示应用目标**：满足赛事答辩、课堂展示、公众科普等使用场景。  
3. **工程实现目标**：在有限时间内保证稳定运行、可维护、可扩展。  
4. **视觉体验目标**：强调东方审美、留白与层次，避免普通后台化展示风格。

### 2.3 设计原则

- **可读优先**：信息层次清晰，图文关系明确。
- **交互克制**：动画和动效服务于叙事，不喧宾夺主。
- **模块解耦**：首页、模块一、模块二、模块三各自可独立迭代。
- **复用导向**：图表页面可通过统一容器重复嵌入。

## 3. 总体技术架构

### 3.1 架构形态

系统采用“React 主应用 + 静态图表页嵌入”混合架构：

- React 负责应用外壳（路由、页面框架、通用样式、业务逻辑）。
- 复杂或历史沉淀图表以独立 HTML 存放在 `public/charts`，通过 `iframe` 接入。
- 不同模块按“功能边界”组织，避免单体页面过度耦合。

### 3.2 架构分层

1. **表现层（UI Layer）**  
   首页、模块入口、内容面板、说明文本、按钮组、图例。
2. **交互层（Interaction Layer）**  
   路由跳转、图表切换、筛选控制、hover/tooltip、局部状态管理。
3. **可视化层（Visualization Layer）**  
   ECharts（图网络、折线、热力、雷达、玫瑰等）与 D3 词云。
4. **静态资源层（Asset Layer）**  
   图片、SVG、地图资源、HTML 图表模板、模块数据文件。

### 3.3 关键架构决策说明

- **决策 A：模块二、三保留 iframe 模式**
  - 原因：历史页面和图表结构已较完整，直接组件化成本高、风险大。
  - 收益：交付效率高，稳定性可控，迁移路径清晰。
- **决策 B：模块一采用 React 组件化图表**
  - 原因：需要多图联动、主题颜色统一、可维护性更高。
  - 收益：可扩展性更好，后续适合进行交互增强。

## 5. 目录结构与职责划分

### 5.1 根目录关键项

- `README.md`：项目总览
- `frontend/`：前端主工程
- `docs/`：项目文档（本目录）

### 5.2 前端目录关键项

- `frontend/src/`：React 业务代码
- `frontend/public/charts/`：模块二、三图表页面及资源
- `frontend/src/modules/module1/`：模块一源码
- `frontend/src/modules/module2/`：模块二容器页
- `frontend/src/modules/module3/`：模块三容器页
- `frontend/src/pages/HeroHome/`：首页视觉与交互

### 5.3 组织规范

- 路由级页面放 `pages` 或 `modules`
- 可复用组件尽量下沉到模块内组件目录
- 静态资源统一放 `public`
- 图表原型资源与业务 React 逻辑隔离

## 6. 功能模块技术说明

## 6.1 首页（Hero）

### 6.1.1 功能目标

- 营造沉浸式首屏氛围
- 提供模块入口
- 建立叙事基调与视觉记忆点

### 6.1.2 技术实现

- React + CSS Modules 管理首屏结构和动画
- 使用透明 PNG/SVG 叠层实现视觉深度
- 使用轻量过渡动画，避免高频重绘

### 6.1.3 关键点

- 标题图层与背景分离，避免文字重复叠加
- 动效控制节奏偏慢，符合文化展示语境
- 首页与模块页视觉风格保持同一色系

## 6.2 模块一（聚落格局）

### 6.2.1 功能目标

- 展示区域分布与数量差异
- 提供时间维度趋势变化
- 支持图形间联动阅读

### 6.2.2 技术实现

- React 组件化封装 ECharts
- 统一主题色配置（如地图热力渐变、折线主色）
- 通过局部状态管理实现联动更新

### 6.2.3 代表能力

- 地图热力颜色梯度映射
- 折线图与信息面板联动
- 图例、标注、提示文案统一管理

## 6.3 模块二（构件专题）

### 6.3.1 功能目标

- 围绕屋顶、马头墙、粉墙、天井等构件进行专题展示
- 每个专题提供多图比较（热力、关系、词云、雷达等）

### 6.3.2 技术实现

- React 容器页加载 `/charts/module2.html`
- 子图页面由 `iframe src` 动态切换
- 词云页面通过 hash 参数渲染对应构件内容

### 6.3.3 关键细节

- 图表字体、底色、标题风格统一
- tooltip、图例、标签颜色可控
- 子图可按专题独立维护，互不干扰

## 6.4 模块三（三雕文化）

### 6.4.1 功能目标

- 展示人物与著作知识卡片
- 提供传承关系图与层级筛选
- 提供工艺流程、纹样、工具、题材、历史等知识面板

### 6.4.2 技术实现

- 主页面通过 `/charts/module3.html` 呈现
- 卡片区支持人物/书籍分类切换
- 关系图区分层级筛选与布局控制两类操作

### 6.4.3 已优化要点

- 节点颜色按层级区分
- 图例补充“代表作节点（空心）”
- 标签颜色统一为黑色，提升可读性

## 7. 数据与资源管理

### 7.1 数据来源形态

- 结构化数组（人物、著作、工艺流程等）
- 图表配置对象（ECharts option）
- 静态图片、SVG 与背景素材

### 7.2 资源管理策略

- 静态公开资源统一放置在 `public`
- 图表相关资源就近放置在 `public/charts`
- 路径使用相对统一规范，降低跨页引用错误

### 7.3 一致性策略

- 主题色统一配置，减少散落魔法值
- 标题文本规范统一（移除“图表X”后缀）
- 图例命名与实际视觉表现严格对应

---

## 8. 交互与可用性设计

### 8.1 交互类型

- 点击：模块切换、选项切换、卡片翻转
- 悬停：tooltip 提示、元素强调
- 滚动：内容浏览、横向卡片滑动

### 8.2 可用性原则

- 操作结果可见（点击后状态明确变化）
- 颜色语义一致（同类节点同色）
- 文字信息清晰（避免低对比灰字）

### 8.3 展示友好性

- 适合投屏演示与课堂讲解
- 首页引导到模块路径清晰
- 模块之间切换逻辑直观

---

## 9. 性能与稳定性

### 9.1 性能策略

- Vite 快速热更新，提升开发效率
- 大图表拆分到独立页面，避免主应用过载
- 资源静态化，减少运行时动态请求复杂度

### 9.2 稳定性策略

- 构建前执行类型检查
- 关键页面采用稳定路径引用
- 常见错误（tooltip undefined、样式不一致）逐项修复

### 9.3 潜在风险

- `iframe` 模式下跨页面联动能力受限
- 图表页多时统一改版工作量较大
- 样式统一依赖规范约束，需持续维护

---

## 10. 工程规范与协作方式

### 10.1 代码规范

- 统一使用 TypeScript
- 保持组件职责单一
- 尽量避免无意义重复样式

### 10.2 协作建议

- 先定义视觉与命名规范，再并行开发
- 模块改动优先“最小影响面”原则
- 对外部仓库合并时采用“目标文件精确比对”流程

### 10.3 质量门禁建议

- 提交前至少执行：
  - `npm run lint`
  - `npm run build`
- 关键页面进行人工回归检查（首页、模块二、模块三）
